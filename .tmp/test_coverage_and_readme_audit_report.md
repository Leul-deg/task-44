## Part 0: `.tmp/` Artifact Lineage

This section gives every artifact in `.tmp/` an explicit provenance record so a reviewer can trace it back to *when* it was written, *what method* produced it, *what source files it read*, and *how it relates to the other artifacts*. Added in response to external feedback that the `.tmp/` artifact lineage was incomplete.

All artifacts in `.tmp/` are **review notes** produced by the repository author during self-audit of the deliverable at `repo/`. They are not generated by any CI pipeline and are not consumed by the application at runtime — they exist purely as human-readable records of audit passes and the fixes that followed. Nothing in `repo/` reads from `.tmp/`.

### 0.1 Artifact map (exactly five files)

```
.tmp/
├── audit_report-01.md                         ← first audit pass (static, no tests run)
├── audit_report-01-fix_check.md               ← per-issue fix record for report 01
├── audit_report-02.md                         ← second audit pass (static + one mvn test run)
├── audit_report-02-fix_check.md               ← per-issue fix record for report 02
└── test_coverage_and_readme_audit_report.md   ← this file: test coverage + README + lineage
```

Producer → consumer relationships:

```
   repo/ source tree
         │
         ▼
 audit_report-01.md ──→ audit_report-01-fix_check.md
         │
         ▼
 audit_report-02.md ──→ audit_report-02-fix_check.md
         │
         ▼
 test_coverage_and_readme_audit_report.md (this file)
         ↳ owns the lineage record (Part 0)
         ↳ owns the test + README audit (Parts 1–5)
         ↳ back-references both fix-checks to avoid double-reporting (Part 0.3)
```

Each numbered audit was written to a blank slate; each fix-check reads *only* its matching audit plus the current `repo/` tree. This report is an independent pass focused on tests and documentation rather than code or architecture.
